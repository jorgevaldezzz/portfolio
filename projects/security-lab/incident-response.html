Incident report

IR-2026-001

Author: Jorge Valdez

Controlled red team exercise covering reverse shell delivery, privilege escalation, credential dumping, and registry-based persistence on a Windows endpoint.

Summary Timeline IOCs Recommendations
Opened 2026-06-05 21:00 UTC · Closed 2026-06-06 02:00 UTC

Executive Summary

Case details

Severity: HIGH. Status: Remediated. Affected host: WIN-NB2M2FMS8NC (10.0.0.20). Affected user: WIN-NB2M2FMS8NC\Administrator. MITRE TTPs: T1059, T1547.001, T1055, T1003.001.

On June 5, 2026, a controlled red team exercise simulated a full attack chain against host WIN-NB2M2FMS8NC. The exercise demonstrated reverse shell delivery via Metasploit, privilege escalation with getsystem, credential extraction from LSASS using Mimikatz, and persistence through a registry Run key pointing to a staged payload at C:\Users\Public\shell.exe.

Detection validation confirmed that custom Wazuh rule 100100 correctly identified the Run key modification at severity level 10. Default rule 92302 also triggered. Analysis identified a detection gap for PowerShell-based registry writes, which bypass Sysmon EID 13 and are covered later in this report.

There was no evidence of lateral movement. All artifacts were purged and affected credentials were reset at the end of the exercise.

Attack Narrative

Initial Access and Payload Delivery

The simulation began with the delivery of a 64-bit Meterpreter payload to the target VM. When executed by the Administrator, a persistent reverse connection was established to the attacker's listener. This phase was logged by Sysmon through process and file creation events.

Privilege Escalation

Using the established session, the operative ran getsystem to obtain SYSTEM-level authority. The technique involved spawning a malicious service host process, which was detected by Wazuh and mapped to MITRE T1050.

Credential Dumping

With elevated privileges, the adversary performed memory forensics on LSASS. The resulting NTLM hash for the Administrator account was successfully harvested. While plaintext credentials were not captured due to modern Windows security defaults, the obtained hash is sufficient for Pass-the-Hash authentication.

Persistence - Registry Run Key

To maintain access, the adversary modified the HKCU registry hive using reg.exe. By creating an Updater value within the Run key, the malicious executable was configured to launch automatically upon user login, aligning with MITRE T1547.001.

This modification generated a Sysmon RegistryEvent (EID 13). Custom Wazuh rule 100100 prioritized the event as level 10, providing higher visibility and specific MITRE tagging compared to the standard alert.

Detection Engineering Notes

Custom Rule Performance

Validation confirms that rule 100100 (Level 10) provides effective detection for reg.exe-based persistence. Rule 100101, designed for LOLBin process matching, remains active but was not triggered by the specific execution flow used in this scenario.

Detection Gap - PowerShell Registry Writes

Gap identified

When using PowerShell's New-ItemProperty cmdlet, no standard Sysmon EID 13 event was generated. As a result, the persistence-specific custom rules were bypassed, and the activity was only captured by generic PowerShell monitoring.

This happens because PowerShell interacts with the registry through the .NET framework, bypassing the specific Win32 API calls monitored by Sysmon for EID 13.

The gap allows an adversary to establish persistence without triggering targeted registry alerts, though broader PowerShell execution alerts may still fire without technical context.

Recommendation: implement supplementary monitoring for Windows Event ID 4104 (PowerShell Script Block Logging). New rules should be authored to detect registry path strings within script content.

Rule Suppression Tuning

Rule 100102 was implemented to filter legitimate environmental activity. Baseline audits identified the following approved processes:

  • VBoxTray.exe - VirtualBox guest additions tray process
  • SecurityHealthSystray.exe - Windows Security Center tray icon

Recommendations

ID Recommendation Detail
R1 Enable Credential Guard The Mimikatz credential dump succeeded because LSASS was unprotected. Windows Credential Guard (requires UEFI + Hyper-V) prevents plaintext credential extraction from LSASS memory.
R2 Restrict PowerShell execution Enable PowerShell Script Block Logging and Constrained Language Mode for non-admin users via GPO to limit payload delivery and registry manipulation via PowerShell.
R3 Supplement Run key detection for PowerShell Author a Wazuh rule chaining off Windows Event ID 4104, matching CurrentVersion\Run in script content, to close the detection gap identified above.
R4 Quarterly suppression review Rule 100102 suppresses known-good updaters. Review the suppression list quarterly as the software inventory changes.
R5 Restrict public staging directories The payload was staged at C:\Users\Public\shell.exe. AppLocker or WDAC should deny execution from C:\Users\Public\ and other world-writable directories.

Appendix - Custom Wazuh Rules

The following custom rules are implemented within the Wazuh manager configuration to enhance detection for the techniques demonstrated in this report. 

<group name="windows,registry,persistence,">

  <!-- Rule 100100: Base persistence detection -->
  <!-- Chains off 92302 (reg.exe Run key write confirmed) -->
  <!-- Elevates severity from level 6 to level 10 -->
  <rule id="100100" level="10">
    <if_sid>92302</if_sid>
    <description>Registry Run key modification detected
      ($(win.eventdata.targetObject))</description>
    <mitre><id>T1547.001</id></mitre>
  </rule>

  <!-- Rule 100101: LOLBin process match (elevated severity) -->
  <!-- Fires when a living-off-the-land binary writes the Run key -->
  <rule id="100101" level="13">
    <if_sid>100100</if_sid>
    <field name="win.eventdata.image" type="pcre2">
      (?i)(powershell|cmd|wscript|cscript|mshta|
      regsvr32|rundll32|certutil|bitsadmin|wmic)\.exe$
    </field>
    <description>High-confidence persistence: Run key written
      by LOLBin ($(win.eventdata.image))</description>
    <group>high_confidence,persistence,lolbin</group>
    <mitre><id>T1547.001</id><id>T1218</id></mitre>
  </rule>

  <!-- Rule 100102: Suppression for known-good processes -->
  <!-- Tuned based on observed Run key writes in this environment -->
  <rule id="100102" level="0">
    <if_sid>100100</if_sid>
    <field name="win.eventdata.image" type="pcre2">
      (?i)(SecurityHealthSystray|VBoxTray)\.exe$
    </field>
    <description>Suppressed: known-good process writing Run key</description>
  </rule>

</group>
