Playbook report

IR Workflow

Author: Jorge Valdez

A repeatable investigation flow for validating alerts, gathering evidence, and deciding when to escalate or close a case.

Analyst guidance · Investigation steps, evidence handling, and escalation notes

Overview

Purpose

The playbook keeps the investigation consistent. It tells the analyst what to check first, what to preserve, and when a case should move up the chain.

This page turns the incident response process into a repeatable workflow. The goal is to reduce guesswork during triage, preserve the evidence that matters, and make escalation decisions based on observable signals rather than instinct alone.

The steps below are designed to work with the same Wazuh, Sysmon, and Windows telemetry used throughout the lab.

First Checks

Start with the alert source, host, user, time, and the process or command data around the event. That gives immediate context before deeper digging begins.

Field What to Verify
Alert source Confirm whether the event came from Wazuh, Sysmon, Windows Security, or another sensor.
Host Check the endpoint name, IP address, and whether the machine is part of the lab's expected scope.
User Note which account was active and whether it matches the expected behavior for that host.
Process Look at the parent-child chain and whether the process path is normal for the activity reported.
Time Anchor the event to a timestamp so related logs can be pulled before and after the alert.
Triage rule

If the alert involves a privileged account, a registry persistence path, LSASS access, or a reverse shell pattern, treat it as potentially malicious until the evidence says otherwise.

Evidence Collection

Keep the process simple. Save the logs, note the timeline, and collect the data that proves why the alert fired.

  • Preserve the triggering alert and any closely related events from the same minute.
  • Capture parent and child processes, command lines, and hashes when available.
  • Export relevant Sysmon events, especially EID 1, 3, 11, 13, and 4104 when PowerShell is involved.
  • Record the user context, affected host, and any registry or file paths tied to the activity.

For this lab, the most useful evidence usually includes a process chain, a file drop, a registry write, and any authentication activity that happened around the same time.

Escalation Criteria

Escalate when the activity matches a known attack pattern, repeats across multiple events, or affects more than one system.

Signal Why It Matters
Registry Run key writes from scripting tools Often indicates persistence, especially when tied to PowerShell, cmd, or another LOLBin.
LSASS access or credential dumping Raises the likelihood of impact beyond simple execution and suggests hands-on-keyboard activity.
Suspicious service creation or privilege escalation Shows the actor is attempting to move from user-level access to deeper control of the host.
Multiple alerts in a short window Correlated events are easier to defend than isolated noise and usually deserve a closer look.

In this lab, escalation is especially warranted if the event maps to T1547.001, T1003.001, or a related attack chain that includes initial access, execution, and persistence.

Follow-up

Close the loop by updating notes, tuning rules if needed, and recording what should change next time.

  • Document the final verdict clearly: benign, suspicious, confirmed malicious, or false positive.
  • Record any tuning changes, suppression logic, or follow-on detections that should be added.
  • Capture recommended containment or recovery steps if the alert represents a real threat.
  • Keep the case notes short enough that another analyst can pick up the thread quickly.

Response Workflow

Stage Action Output
1. Validate Confirm the alert is real and not a noisy test or normal admin action. A trusted starting point for the investigation.
2. Scope Check nearby logs, related alerts, and the process chain leading into the event. A clearer picture of what happened before and after.
3. Preserve Save the relevant telemetry before it rolls off or gets overwritten. Evidence that can support analysis or reporting.
4. Decide Determine whether to escalate, contain, or close based on the evidence. A documented and defensible case outcome.
